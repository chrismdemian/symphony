# Scenario 3O.2 — `open_pr` (PR generation)

**Gate 3 production scenario for Phase 3O.2.**

Drives the real orchestrator MCP server (`startOrchestratorServer` + an
in-memory MCP client) end-to-end through the `open_pr` tool against a **real
git repo + bare remote**. The two external processes are isolated:

- **`claude` is real.** PR title/description are generated by a real
  `claude -p --output-format json` subprocess (the default one-shot runner)
  over the worker branch's real diff + commits. Skips gracefully when
  `claude --version` is unavailable.
- **`gh` is faked.** A stub `GhRunner` (injected via the `openPrGhRunner`
  server seam) records the `gh pr create` call and returns a canned URL.
  **No real pull request is ever opened.** The real `git push` to the local
  bare remote DOES run (no outward effect).

## Assertions

1. **Tier-1 capability gate (deterministic, no claude).** `open_pr` carries
   the `external-visible` capability flag → Tier ≥2 floor. At `initialTier: 1`
   the dispatch shim denies the call before the handler runs; the fake `gh`
   records zero `createPr` calls.
2. **Happy path (real claude).** At `initialTier: 2`, after a worker commits a
   change on its worktree branch, `open_pr`:
   - pushes the branch to the bare remote,
   - generates a non-empty title + markdown body via real claude,
   - calls the fake `gh pr create` with `base=main`, `head=<worker branch>`,
     and the generated title/body,
   - returns the canned PR URL with `description_source` of `llm` (or
     `heuristic` if claude's output didn't parse — both are acceptable; the
     assertion is that real content reached `gh`).
