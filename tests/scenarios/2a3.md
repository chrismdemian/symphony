# Phase 2A.3 — Production Scenario

Drive the full project/task/worktree tool chain through an in-process
MCP client against a Symphony orchestrator backed by a real `git`
worktree stack. No `claude -p` subprocess — this scenario is pure
orchestration-layer, covering Maestro's plan-then-act flow for
enqueuing work and materializing isolated worktrees.

## Given

- A fresh temp git repo with one commit (no `claude` required).
- An orchestrator server created via `InMemoryTransport` with:
  - `initialMode: 'plan'`
  - `defaultProjectPath: <temp repo>`
  - a real `WorktreeManager`
  - a WorkerManager stub (no spawns in this scenario)
  - in-memory `ProjectRegistry` + `TaskRegistry`
- Two registered projects: the temp repo (as `main`) and a second bare
  name used to exercise name→path resolution.

## When

1. Client calls `list_projects`. Both names should appear.
2. Client calls `get_project_info` for `main`. The response surfaces the
   project path, confirming name→path resolution works through MCP.
3. Client calls `create_task` twice in PLAN mode against `main` —
   Maestro decomposes a two-step plan before user approval.
4. Client calls `list_tasks` with `status=pending`. Both task ids
   appear.
5. Client switches orchestrator mode to ACT. Client calls
   `create_worktree` for `main`. The `WorktreeManager` materializes a
   real linked worktree under `<repo>/.symphony/worktrees/wt-*`.
6. Client calls `update_task` → `in_progress` on task 1 (with notes).
7. Client calls `update_task` → `completed` on task 1 with a result.
8. Client calls `update_task` → `cancelled` on task 2 (pending → cancelled).
9. Client calls `list_tasks` again with no filter. All tasks should be
   in their expected terminal or progressed states.
10. Client attempts an invalid transition (`completed` → `failed`) and
    expects `isError: true` from the tool shim (state machine enforced).

## Then

Observable state:

- `list_projects` structuredContent.total = 2.
- `get_project_info` returns the path matching the temp repo.
- Two task snapshots have `status: 'pending'` after creation.
- After ACT-mode `create_worktree`, a directory exists on disk under
  `<repo>/.symphony/worktrees/wt-<hex>`; `git worktree list --porcelain`
  for the temp repo includes the new path.
- Task 1 finishes at `completed` with a defined `completedAt` and the
  supplied result string. Task 2 ends `cancelled`.
- Terminal-task transition attempt returns `isError: true` with an
  "Invalid transition" message.

## Cleanup

Worktree removed via `WorktreeManager.remove`, temp repo removed via
`rmSync({ force: true, recursive: true })`. Orchestrator closed
(`server.close()`). No orphan processes or leftover worktrees.
