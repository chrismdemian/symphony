# Phase 3A scenario — TUI launcher integration

## Given

- A fresh `home` directory with no Maestro workspace yet.
- A fake `MaestroProcess` that completes `start()` synchronously and emits a `system_init` event.
- A fake `RpcClient<SymphonyRouter>` that returns `[]` for `projects.list` / `workers.list` and `{mode: 'plan'}` for `mode.get`.
- A fake `stdout` stream with `isTTY: true` and `columns: 120`, `rows: 30` so `runTui` engages instead of falling back to readline.

## When

1. Call `runStart({ ... })` with the above fakes.
2. Wait for the launcher to settle (Maestro started, hook installed, TUI rendered).
3. Call `handle.stop('scenario shutdown')`.
4. Await `handle.done`.

## Then

- `runStart` returns `{stop, done}` without throwing.
- The fake Maestro's `start()` is called exactly once — no double-start.
- The Stop hook is installed in the workspace's `.claude/settings.local.json`.
- After `stop()`, the hook is uninstalled (settings file no longer mentions `SYMPHONY_HOOK_PORT`).
- The fake Maestro's `kill()` is called exactly once during teardown.
- The RPC client's `close()` is called.
- `handle.done` resolves cleanly within 10 s.
- No orphan timers / handles keep the test process alive past resolution.

## Why this is the right shape

The 2C.2 scenario test (`tests/scenarios/2c2.test.ts`) composes pieces directly (`MaestroProcess` + `MaestroHookServer` + `WorkerManager`) rather than spawning `symphony start` as a subprocess. Phase 3A follows the same pattern:

- `runStart` IS the integration boundary — wiring it up in-process with fakes exercises every seam (RPC bootstrap, hook server, Maestro spawn, TUI engagement, cleanup chain).
- A subprocess spawn would inherit the test runner's stdout (non-TTY in vitest), so the TUI would always fall back to readline — no actual TUI render coverage.
- The existing 3 `runStart` unit tests (`tests/cli/start.unit.test.ts`) prove the readline fallback still works post-3A. This scenario proves the TUI engagement path works.
