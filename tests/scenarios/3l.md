# Phase 3L scenario — Task Queue Panel

End-to-end test of the queue panel surfacing pending tasks in the
workers panel.

## Given

- Boot `runStart` against a fake RPC client.
- Fake `queue.list` returns 3 pending entries across 2 projects, sorted
  by `enqueuedAt`:
  1. `r1` — `add filters` (MathScrabble)
  2. `r2` — `fix scraper timeout` (CRE Pipeline)
  3. `r3` — `update auth` (MathScrabble)
- Fake `workers.list` returns one running worker so the panel has both
  active workers AND queued tasks visible.

## When

Drive the TUI through three keystrokes:

1. Press `Tab` (focus → workers panel)
2. Press `j` once (cursor advances from default worker to queue-header)
3. Press `j` again (cursor advances to first queue row `r1`)
4. Press `X` (cancel queued)
5. Wait for next poll tick.

## Then

- `queue.list` RPC was called ≥ 1 time
- Rendered frame contains `Queue (3 pending)` header
- Rendered frame contains `Next →` marker on the head row
- Rendered frame contains `add filters`, `fix scraper timeout`, `update auth`
- Rendered frame contains `(MathScrabble)` and `(CRE Pipeline)` parens
- After pressing `X` on the selected r1 row, `queue.cancel` was called
  with `{ recordId: 'r1' }`
- The success toast `cancelled queued: add filters` appears
