# Phase 5A — Production scenario: `.symphony.json` `project` section persists to ProjectRecord

## Given

- Empty Symphony state dir (fresh `:memory:` SQLite via the SQLite project store).
- Two project directories on disk, both containing `.symphony.json`:
  - `proj-full/` — file's `project` section sets ALL 9 Phase 5A fields plus several existing
    `ProjectConfigInput` fields (`testCommand`, `verifyCommand`, etc.).
  - `proj-broken/` — file's `project` section contains an unknown key (`qualityPipelin: 'full'`)
    so Zod validation fails.
- Third project `proj-override` has a `.symphony.json` setting `qualityPipeline: 'simplified'`
  AND a caller-supplied `options.projectConfigs[proj-override].qualityPipeline = 'full'`.

## When

Programmatically invoke `mergeProjectConfigsWithFiles(projects, callerConfigs)` (the same helper
`startOrchestratorServer` uses at boot) for the three projects, then `store.register(...)` each
merged record into a `SqliteProjectStore`.

## Then

1. **`proj-full`** — `store.get('proj-full')` returns a `ProjectRecord` with every Phase 5A
   field populated from the file. `console.warn` is NOT called.
2. **`proj-broken`** — `console.warn` is called once with a message matching `failed validation`.
   `store.get('proj-broken')` returns a `ProjectRecord` with NO Phase 5A fields set
   (`qualityPipeline` is undefined; same for the rest). Boot does not crash.
3. **`proj-override`** — `store.get('proj-override').qualityPipeline === 'full'` (caller wins
   over file).

## Notes

- This scenario exercises the public path: `mergeProjectConfigsWithFiles` (exported) +
  `SqliteProjectStore.register` (the same store `startOrchestratorServer` constructs).
  `seedProjectsFromMap` is module-private; the end-to-end through `startOrchestratorServer`
  is tested by the regression sweep (every prior scenario boots an orchestrator and the
  `.symphony.json` loader runs without error).
- The drift-lock integration test (`5a-config-drift`) is part of the same Sub-Phase
  Acceptance gate but lives outside `tests/scenarios/` because it asserts schema/doc
  symmetry, not user-observable behavior.
