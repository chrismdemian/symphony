# Phase 7B.3 scenario — host enrichment end-to-end

## Goal

End-to-end verification that the host consumes the three SDK `_meta`/event
seams: event-handler tools stay out of Maestro's toolbelt, an over-permissioned
tool is refused (fail-closed), and the new create/spawn events are sourced and
delivered to subscribed plugins — all through a real orchestrator boot + a real
plugin subprocess.

## Tested paths

1. **Two-gate activation + real subprocess.** Orchestrator boots with
   `options.plugins.enabled` AND the `pluginsEnabled` config switch ON. The real
   `enrich` plugin (`tests/fixtures/plugins/enrich/server.mjs`) is spawned via the
   production StdioClientTransport. Its tools carry real MCP `_meta` markers.

2. **Hidden event-handler tools (deliverable 1).** A Maestro-equivalent MCP
   `Client`'s `listTools()` does NOT contain `enrich__on_task_created` or
   `enrich__on_worker_spawned` — they are notification sinks the host calls, not
   tools Maestro should call. They remain dispatchable (proven below).

3. **Per-tool permission enforcement (deliverable 3).** `enrich__safe_read`
   (needs `task:read`, granted by the manifest) IS in the toolbelt;
   `enrich__over_reach` (needs `task:write`, NOT granted) is REFUSED and absent —
   the offending tool only, never the whole plugin (per-tool isolation).

4. **New event source delivered end-to-end (deliverable 2).** Calling the real
   `create_task` MCP tool fires `onTaskCreated` through the full chain
   (`TaskStore.create` → `fanOutTaskCreated` → `pluginHost.dispatchEvent` → the
   plugin's hidden `on_task_created` handler). Observed via the plugin's
   `get_event_count` advancing 0 → 1.

5. **Clean teardown.** `handle.close()` shuts the plugin subprocess down with no
   leaked process.

## Coverage notes

- `onWorkerSpawned` end-to-end (server fan-out → delivery) is covered by the unit
  test (`worker-lifecycle-onspawned`, hook fires once on spawn) + the integration
  test (`7b3-host-enrichment.integration`, `dispatchEvent('onWorkerSpawned')`
  reaches the hidden handler). A scenario spawn would require a real `claude -p`
  subprocess (the 7A scenario deliberately uses `echo` for the same reason).
