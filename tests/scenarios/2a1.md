# Phase 2A.1 — Production Scenario

A real `claude -p --strict-mcp-config --mcp-config <json>` worker, with Symphony's orchestrator MCP server as its sole MCP provider (stdio), is instructed to call the `think` tool. Symphony's MCP server receives the call through the full dispatch chain (safety guard → capability evaluator → mode gate → handler) and returns a structured result. The worker's stream emits `tool_use` for the MCP tool and `tool_result` with the server's response. No `parse_error` events.

Composes every prior scenario: 1A's stream parsing, 1B's worker manager (trust, env allowlist, stdin close on result), 1C/1D's worktree + preservation + pool work — but the new ground is the end-to-end MCP dispatch. This is the first time Symphony's own MCP server drives a real worker.

## Given

- A `claude` binary resolvable via PATH (`claude --version` exits 0). If missing, this scenario **skips with a warning** so CI without claude still passes.
- Symphony's built CLI at `dist/index.js` (produced by `pnpm build`).
- A fresh temp git repo with one commit (no special files required — this scenario tests the MCP server, not worktrees).
- An MCP config JSON file written to the repo pointing at `node <symphony-dist>/index.js mcp-server` as the `symphony` server.

## When

1. Caller ensures `dist/index.js` is up to date (runs `pnpm build` as setup).
2. Caller writes `.mcp.json` with `{ "mcpServers": { "symphony": { "command": "node", "args": [<absolute dist path>, "mcp-server"] } } }`.
3. Caller spawns `claude -p --strict-mcp-config --mcp-config .mcp.json --permission-mode bypassPermissions --output-format stream-json --verbose` in the repo dir.
4. The prompt sent on stdin instructs the worker to: (a) call the `think` tool on the `symphony` MCP server with a one-shot ledger JSON, (b) after `tool_result` returns, emit a structured completion JSON fence echoing the fact that the tool was invoked.
5. Caller parses the worker's stdout via Phase 1A's stream parser.
6. Caller waits for the worker to exit.

## Then

Observable state:

- Worker exits cleanly (`status=completed`, `exitCode=0`).
- Stream contains at least one `system_init` event whose `mcpServers` array includes `{ name: "symphony", status: "connected" }` (or equivalent "ready" status string Claude uses) — proves the MCP handshake completed through our stdio transport.
- Stream contains at least one `tool_use` event whose `name` contains `symphony` AND `think` (Claude Code prefixes MCP tools, typical form `mcp__symphony__think`).
- Stream contains at least one `tool_result` event whose text content references the successful dispatch (`ledger recorded`) — proves our dispatch shim ran end-to-end.
- Stream contains at least one `result` event; it has a non-empty `session_id`.
- Stream contains zero `parse_error` events.
- Structured completion event is emitted and its `audit` field is `"PASS"`.

## Cleanup

Temp repo and MCP config removed. No leftover processes: dispatched MCP server is a child of claude, which closes it on exit.
