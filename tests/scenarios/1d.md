# Phase 1D — Production Scenario

A `worktreePool.enabled: true` project routes `WorktreeManager.create(...)` through `WorktreePool.claimReserve(...)` instead of the synchronous 3-7 second `git fetch + git worktree add` path. The claimed worktree is indistinguishable from a sync-created one: identical branch, same preserved gitignored files, same `.git/info/exclude` contents. A real `claude -p` worker spawned inside it reads a preserved `.env` and emits a structured completion. After claim, the pool replenishes another reserve in the background.

Composes Phase 1C's scenario (real worktree + real claude reads preserved `.env`) and extends it with the pool path.

## Given

- A `claude` binary is resolvable via PATH (`claude --version` returns 0). If not, this scenario **skips with a warning** so CI without claude still passes.
- A fresh temp directory containing an initialized git repo with one commit and a `.gitignore` that excludes `.env`.
- A `.env` file written to the project root after the initial commit (so it is gitignored, not tracked).
- A `.symphony.json` at the project root declaring `{"worktreePool": {"enabled": true}}`.
- Phase 1D `WorktreePool` + Phase 1C `WorktreeManager` wired together with a shared `ProjectLockRegistry`.

## When

1. Caller initializes the temp git repo (`git init`, one commit) and writes `.env` with a known secret to the project root.
2. Caller writes `.symphony.json` enabling the pool.
3. Caller constructs a single `ProjectLockRegistry`, `WorktreePool` (runPoll: false), and `WorktreeManager` sharing the registry, with a `WorkerManager` for the worker spawn.
4. Caller awaits `pool.ensureReserve(projectPath)` so a reserve is warm.
5. Caller invokes `WorktreeManager.create({ projectPath, workerId, shortDescription })` and captures the `onPoolClaim` event to confirm the claim path was taken.
6. Caller spawns a `claude -p` worker in the claimed worktree with a prompt that reads `.env` and emits a structured completion echoing the secret.
7. Caller drains the worker stream and awaits exit.
8. Caller confirms the pool has warmed a *new* reserve in the background via `pool.ensureReserve(projectPath)` + `listReserves()`.
9. Caller invokes `WorktreeManager.removeIfClean(info.path)` to clean up.
10. Caller shuts down the worker manager and pool.

## Then

Observable state:

- The `onPoolClaim` event fires exactly once for the target worker ID (proves the claim path ran, not the sync fallback).
- `info.path` equals `<projectPath>/.symphony/worktrees/<workerId>` and exists on disk.
- `info.branch` equals `symphony/<workerId>/<slug>` (reserve's `_reserve/<hash>` branch was renamed).
- `<info.path>/.env` exists and its content is byte-equal to the source `.env`.
- `<info.path>/.git/info/exclude` contains every default git-exclude pattern (`.agent_context`, `CLAUDE.md`, `AGENTS.md`, `.symphony/`, `.claude/`).
- Worker exits with `status === 'completed'` and `exitCode === 0`.
- Worker emits exactly one `structured_completion` event whose `report.did[0]` contains the known secret value.
- No `parse_error` events.
- After the claim, a fresh reserve exists in `pool.listReserves()` whose `path` is NOT the claimed path (background replenishment succeeded).
- `removeIfClean(info.path)` returns `true` and the worktree directory is gone.
- The branch the worktree was on no longer appears in `git branch --list`.

## Ground Truth Is Observable

Every assertion derives from Symphony event hooks, filesystem state, event-stream output, or `git` command output. The worker's textual reasoning is irrelevant — only the structured completion's `did` array is consulted.

## Gates

- **Skip-gracefully**: If `claude --version` fails (non-zero exit or missing binary), the test is marked `skip` with the reason logged. `pnpm verify` still passes.
- **Timeout budget**: 90 seconds for the worker turn (cold-start friendly).
- **Network dependency**: Real claude spawns talk to Anthropic's API. Running this scenario locally consumes a small amount of Max-subscription quota — acceptable for a single acceptance check.
- **Regression**: 1A + 1B + 1C scenarios must still pass before this one. `vitest.scenarios.config.ts` runs them in order.
