# Phase 8A scenario — Notion sync end-to-end (real DB, real server)

## Goal

End-to-end verification of the in-tree Notion connector against a real
orchestrator boot backed by a temp-FILE SQLite database (not `:memory:`), so
external-link persistence is proven across a fresh store instance — the
restart-survives-dedup guarantee that makes re-syncing safe.

A FAKE Notion connector is injected (no live API), but everything else is
production: real `startOrchestratorServer`, real `SqliteTaskStore` /
`SqliteExternalLinkStore`, real MCP `Client` dispatch through the capability
shim (tier 2, since `sync_notion` declares `requires-secrets-read` /
`requires-network-egress-uncontrolled` / `external-visible`).

## Tested paths

1. **Tool surface + capability gating.** `sync_notion` is registered only
   because a connector is injected, and a tier-2 MCP `Client` can call it
   (the three capability flags require tier >= 2).

2. **Create + route + link.** Three Notion pages (two open routed to two
   different projects, one already `Done`) → two Symphony tasks created in
   SQLite, two external links written, the done page skipped.

3. **Idempotent re-sync.** A second `sync_notion` over the same pages creates
   zero tasks (`skippedExisting === 2`) — dedup by external link.

4. **Persistence across a fresh store.** After closing the server, a brand-new
   `SqliteExternalLinkStore` opened over the SAME db file still resolves the
   links — proving re-sync after a Symphony restart stays idempotent (the
   cross-instance reload guarantee, mirroring 7C's plugin-store reload).

5. **Bidirectional writeback.** Driving an imported task to `completed` fires
   the `onTaskStatusChange` fan-out → the Notion writeback hook → the
   connector's `writeBackStatus(pageId, 'completed')`. A task with no Notion
   link triggers no writeback.

## Coverage notes

- The connector's own Notion I/O (data-source resolution, pagination, property
  mapping, status-vs-select writeback shape, the 3 req/s throttle) is covered
  by `tests/integrations/notion.unit.test.ts` against a hand-written fake
  client.
- The `:memory:` variant of paths 1–3 + 5 lives in
  `tests/integration/8a-notion-sync.integration.test.ts`; this scenario adds
  the temp-FILE persistence leg (path 4).
