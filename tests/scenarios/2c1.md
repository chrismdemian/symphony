# Phase 2C.1 — Maestro Process + MCP Attachment

## Production Scenario

**Subject under test:** `MaestroProcess` boots a `claude -p` long-lived
subprocess wired to Symphony's MCP server via `--mcp-config`, with a
deterministic session UUID, multi-turn stdin chat, and typed event stream.

**Acceptance:**

1. The parent (`MaestroProcess` host) writes Maestro's CLAUDE.md and the
   `--mcp-config` JSON file to `~/.symphony/maestro/` from the v1 prompt
   artifact + provided template variables.
2. The parent connects to the orchestrator-server child's WS-RPC and
   successfully calls `projects.list` — proves Phase 2B.2 wiring is intact
   and `{registered_projects}` resolution will work in Phase 2C.2's launcher.
3. `MaestroProcess.start()` resolves once `system_init` arrives (proves the
   subprocess is up and the MCP server was found).
4. `MaestroProcess.sendUserMessage()` writes a stream-json `user` chunk
   into Maestro's stdin and Maestro emits assistant_text + tool_use +
   tool_result + turn_completed events through the typed pipeline.
5. `MaestroProcess.kill()` shuts the subprocess down cleanly.

**Real-Claude gate:** when `claude --version` succeeds, an additional test
runs the full integration with the real Claude CLI calling a real MCP tool
(`list_workers`) on the orchestrator-server child. Skipped on CI without
Claude installed; required for local Sub-Phase Acceptance Gate 3.

**Regression coverage:**
- Phase 1B `keepStdinOpen` + `disableTimeout` semantics.
- Phase 2A.4a `list-projects` MCP tool reachable via RPC.
- Phase 2B.2 RPC descriptor file format + Bearer token auth.
