# Phase 2C.2 — Production scenario

> **Sub-Phase Acceptance Gate 3**. Exercises the full Stop-hook + session-resume
> + `symphony start` plumbing end-to-end on a real `claude -p` subprocess.

## Always-run path

Spawns the bootstrap mcp-server (`tsx src/index.ts mcp-server --in-memory ...`),
constructs `MaestroHookServer` + `MaestroProcess` directly with a fake-claude
spawner, sends a user message, and asserts:

1. `installStopHook` writes `SYMPHONY_HOOK_PORT` into the workspace's
   `.claude/settings.local.json`.
2. A POST to the hook server (token + event headers correct) lands a `200`
   response and surfaces an `idle` event on `MaestroProcess.events()` with
   the parsed payload.
3. `uninstallStopHook` strips the entry.
4. The `SYMPHONY_HOOK_PORT` and `SYMPHONY_HOOK_TOKEN` env vars round-trip
   into the spawned worker's environment (verified via the fake-claude
   `env-echo` scenario).

Always-run path uses `fake-claude-interactive.mjs` so it never depends on
the real Claude binary.

## Real-Claude path (opt-in)

Set `SYMPHONY_2C2_REAL_CLAUDE=1` to enable. Skips otherwise.

1. **Boot 1**: `runStart()` spawns the bootstrap + real Maestro. Send a
   single user message ("respond with the literal word OK and nothing
   else"). Wait for an `idle` event (proves the Stop hook hit our server).
   Capture `system_init.sessionId`.
2. **Graceful shutdown**: `handle.stop()` → settings.local.json no longer
   contains `SYMPHONY_HOOK_PORT`.
3. **Boot 2**: re-run `runStart()` with the same `home`. Capture
   `system_init.sessionId`. Assert it equals boot 1's. Proves
   `MAESTRO_SESSION_UUID` resume works through the launcher.

## Gates passed by this scenario

- Gate 3 (production scenario, real-Claude binary): the real-Claude path.
- Gate 4 (regression, no breakage of prior phases): the always-run path
  exercises every primitive 2C.1 already validated, plus the Stop-hook
  additions, with no new mocks beyond fake-claude.
