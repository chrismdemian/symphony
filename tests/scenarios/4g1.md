# Phase 4G.1 scenario — audit loop + iterate-in-place + retry cap

## Goal

End-to-end verification that the production `audit_changes` MCP tool path
auto-bumps `WorkerRecord.auditAttempts` server-side, surfaces the new
value to Maestro via `AuditResult.auditAttempts` (and the MCP response's
`structured_content.audit_attempts`), and lets Maestro detect the
3-FAIL cap by reading that value rather than tracking state itself.

Ground Truth Is Observable: assertions are on emitted MCP response
content + the persisted/in-memory counter on the WorkerRecord. The
counter is bumped by the REAL `runAudit` core; only the `claude -p`
reviewer subprocess and the worktree git repo are stubbed/seeded.

## Given

- A real `WorkerRegistry` + a real ProjectRegistry.
- A real worktree (a temp git repo with one staged change so
  `diffWorktree` returns non-empty content).
- The REAL `makeAuditChangesTool` MCP tool wired against the registry,
  invoked via the same dispatch surface Maestro uses.
- A stub `OneShotRunner` that returns a FAIL verdict twice then a PASS
  — simulating a typical iterate-in-place loop.

## When / Then

### Section 1 — counter advances + surfaces in MCP response

Invoke `audit_changes(worker_id)` three times in sequence.

- First call: `verdict=FAIL`; response text contains `attempt: 1`;
  `structured_content.audit_attempts === 1`.
- Second call: `verdict=FAIL`; text contains `attempt: 2`;
  `audit_attempts === 2`.
- Third call: `verdict=PASS`; text contains `attempt: 3`;
  `audit_attempts === 3`. (Counter does not reset on PASS — by
  design, see audit-loop-constants.ts JSDoc.)
- After all three calls, `registry.get(workerId)?.auditAttempts === 3`.

### Section 2 — store error does not break the audit

Sabotage `registry.bumpAuditAttempts` to throw. Invoke `audit_changes`.

- Response still returns a valid verdict.
- `structured_content.audit_attempts` is absent (the bookkeeping
  failed; the audit is still authoritative).
- The MCP tool does NOT return `isError: true` for the bump failure —
  the audit's verdict drives `isError` per spec.

## Acceptance

Pass condition: all Section 1–2 assertions hold via the real
`makeAuditChangesTool` + real `runAudit` + real `WorkerRegistry`.
Deterministic; no real `claude -p` required (the reviewer's verdict
is produced by a stub `OneShotRunner` returning a hand-shaped
result — the standard Track-1 boundary).
