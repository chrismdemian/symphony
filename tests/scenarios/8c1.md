# Scenario 8C.1 — Linear sync (foundation + connector)

**Goal:** prove the Linear connector end-to-end against a real orchestrator
server, a real `LinearConnector` (over the injected `LinearClientLike` seam, no
live API), and temp-file SQLite — plus the persistence-across-a-fresh-store leg
and the on-the-wire status writeback.

**Setup**
- Temp-file SQLite db.
- A fake `LinearClientLike` returning a mix of open + terminal issues across two
  teams, plus workflow states for writeback resolution.
- A real `LinearConnector` built on that client, injected via `linearConnector`.
- Two registered projects (`alpha`, `beta`) so routing-by-team resolves.
- Server booted at autonomy tier 2 (`sync_linear` is secrets/network/external-visible).

**Steps & expectations**
1. `sync_linear` creates one pending task per open issue, routes by team key,
   and **skips** issues already in a completed/canceled state (`skippedDone`).
2. A second `sync_linear` creates nothing (idempotent — `skippedExisting`).
3. External links persist: a fresh `SqliteExternalLinkStore` over the same db
   file lists every imported issue id under `source='linear'`.
4. Completing a task fires the writeback hook → the real connector resolves the
   team's first completed-type workflow state and calls `updateIssueState` on
   the client with that state id.
