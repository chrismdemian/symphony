# Phase 5C — Production scenario: `task_notes` MCP tool externalizes progress notes

## Given

- A real git project directory at `<sandbox>/proj/` with `git init` + initial commit.
- A real on-disk SQLite DB.
- The project registered via `SqliteProjectStore.register` (same call path
  `startOrchestratorServer` uses at boot).
- A real `SqliteTaskStore` wired with the SAME `onNotesAppended` callback shape
  `server.ts` constructs — `(snapshot) => mirrorTaskNotes({projectPath, taskId, notes})`.
- Three real tasks: two will receive notes, one stays empty.
- `task_notes` + `list_tasks` MCP tools from `makeTaskNotesTool` / `makeListTasksTool`
  (the same factories `server.ts:registerTaskNotesTool` registers).

## When

A user-style scenario unfolds against the live wiring:

1. Maestro calls `task_notes(action: "append", task_id: A, text: "started work")`.
2. Maestro calls `task_notes(action: "append", task_id: A, text: "found the cause")`.
3. Maestro calls `task_notes(action: "append", task_id: B, text: "ready to ship")`.
4. The mirror writer (real `fs.promises`) gets a chance to flush.
5. Maestro calls `task_notes(action: "read", task_id: A)`.
6. Maestro calls `task_notes(action: "list")` (no project filter).
7. Maestro calls `list_tasks` with default options.
8. Maestro calls `list_tasks` with `include_notes: true`.

## Then

1. **After step 4 (mirror flush)**:
   - `<sandbox>/proj/.symphony/tasks/<A>/notes.md` exists, has 0o644 mode (default),
     contains the AUTOGENERATED banner + two `## YYYY-MM-DD HH:MM:SS UTC` sections
     with "started work" and "found the cause" bodies.
   - `<sandbox>/proj/.symphony/tasks/<B>/notes.md` exists, contains one section
     with "ready to ship".
   - The third task's task-id directory does NOT exist (no mirror on zero-notes).

2. **After step 5 (`task_notes read`)**:
   - `res.isError` is false.
   - The text content contains both note bodies in order.
   - `structuredContent.total === 2`, `structuredContent.returned === 2`,
     `structuredContent.truncated === false`.

3. **After step 6 (`task_notes list`)**:
   - `structuredContent.total === 2` (two tasks with notes).
   - The summaries cover tasks A and B; the empty third task does NOT appear.
   - Each summary carries `count`, `firstAt`, `lastAt`.

4. **After step 7 (`list_tasks` default)**:
   - `structuredContent.notesIncluded === false`.
   - Every entry in `structuredContent.tasks[]` has NO `notes` key (the field is
     stripped, not just empty).

5. **After step 8 (`list_tasks include_notes:true`)**:
   - `structuredContent.notesIncluded === true`.
   - Task A's entry carries a `notes` array of length 2 with both note texts.
   - Task B's entry carries one note.
   - The third task's entry carries an empty `notes` array.

## Notes

- Real fs, real SQLite, real registered project. No mocks for fs/SQL paths.
- The mirror writer is async (Promise micro-task → fs/promises macro-task);
  the scenario uses a small `setTimeout` budget to let the rename land before
  asserting on disk.
- Mirror failure is best-effort by design (see `mirrorTaskNotes` JSDoc). This
  scenario covers the happy path; the unregistered-path / fs-write-failure
  edges are covered by the integration test (`5c-task-notes.integration.test.ts`).
- The integration test exercises the same wiring at a finer-grained API; this
  scenario verifies the user-observable shape end-to-end.
