# Phase 3P scenario — cross-project task dependencies through the launcher

## Goal

End-to-end verification that `task_ready` events fire as chat system rows AND
that the `/deps` slash command surfaces the dep-graph popup, both driven through
`runStart` with a fake RPC. Mirrors the 3O.1 scenario shape exactly.

## Tested paths

1. **`task_ready` event → chat row.** Fake RPC's `subscribe('task-ready.events')`
   fires a canned `TaskReadyEvent`. After the first poll cycle, the chat panel
   contains a system row with:
   - `✓` glyph (mapped from `statusKind='completed'`)
   - "Task ready:" headline
   - Project name visible

2. **Cross-project `task_ready`.** Same as above but `task.projectId !==
   unblockedBy.projectId`. The headline includes both project names; the row
   still uses the gold ✓ glyph.

3. **`/deps` slash command opens the popup.** User types `/deps` + Enter into
   the chat InputBar. The popup mounts and renders the graph snapshot returned
   by `tasks.graph()` (fake RPC returns a fixed chain).

4. **`/deps` typed without graph data renders the empty-state hint.** Fake RPC
   returns `{nodes:[], edges:[], cycles:[]}`; popup shows the "No task
   dependencies yet…" message.

## Fake RPC shape

In addition to the baseline `makeFakeRpc()` shape from earlier scenarios:

- `subscribe('task-ready.events', ...)` listener — fire canned events via
  `setImmediate(() => events.forEach(listener))`.
- `tasks.graph()` — return a `TaskGraph` snapshot the test wants the popup to
  render.

## Out of scope

- The dispatcher's emit logic (covered by
  `tests/orchestrator/task-ready-dispatcher.unit.test.ts`).
- The store's filter (`tests/state/sqlite-task-store-3p.unit.test.ts`,
  `tests/state/task-registry-3p.unit.test.ts`).
- Cycle detection (`tests/orchestrator/task-deps.unit.test.ts`).
