# Phase 4F.1 scenario — custom droid spawn installs an enforcing PreToolUse fence

## Goal

End-to-end proof that a custom droid spawned through the REAL MCP
`spawn_worker` handler produces a worktree whose REAL
`settings.local.json` PreToolUse command — driven by the REAL policy env
the same spawn wired — actually ENFORCES the droid's `tools_allowed` /
`tools_denied` / `write_paths` when executed exactly as Claude Code
would execute it.

Symphony spawns every worker with `--permission-mode bypassPermissions`,
under which Claude Code's own `--allowedTools`/`--disallowedTools` are
no-ops; the PreToolUse hook is the only enforcement that survives bypass
(plan decision). This scenario closes the loop: spawn → installed
command + env → real hook subprocess → real exit code.

Ground Truth Is Observable: assertions are on the bytes written to
`settings.local.json`, the env on the captured `WorkerConfig`, and the
real exit code + stderr of the real fence-hook artifact — never on a
self-report. The ONLY stub is the `claude` subprocess
(`WorkerManager`), the standard Track-1 boundary.

## Given

- A real git project sandbox with
  `<project>/.symphony/droids/locked-droid.md` declaring
  `tools_allowed: [read, grep, glob, write]`,
  `tools_denied: [bash, edit]`, `write_paths: ["DESIGN.md"]`.
- REAL `WorktreeManager`, REAL `WorkerRegistry`, REAL
  `createWorkerLifecycle`, REAL `makeSpawnWorkerTool` handler, REAL
  droid registry/parser/settings-writer/fence-hook.
- The fence-hook is invoked via the EXACT command
  `buildDroidFenceHookCommand()` writes into `settings.local.json`,
  with the EXACT `extraEnv` the spawn put on the `WorkerConfig`, and a
  crafted PreToolUse JSON on stdin — i.e. precisely how Claude Code
  drives it.

## When / Then

### Section 1 — spawn wires the fence

Spawn `role: "locked-droid"` through the real handler.

- `<worktree>/.claude/settings.local.json` exists; its
  `hooks.PreToolUse[].hooks[].command` is the resolver-built command and
  carries the `--symphony-droid-fence` marker; it is NOT suffixed with
  `|| true`.
- The captured `WorkerConfig.extraEnv` carries `SYMPHONY_DROID_FENCE`
  (canonical policy JSON) + `SYMPHONY_DROID_WORKTREE` (= worktree path),
  and `allowExtraEnvKeys` exempts both.

### Section 2 — the installed fence actually enforces (real subprocess)

Run the installed command with the installed env + a PreToolUse payload:

- `Bash` (in `tools_denied`) → exit 2, stderr names the denial.
- `Read` (in `tools_allowed`) → exit 0.
- `Write` to `<worktree>/DESIGN.md` (in `write_paths`) → exit 0.
- `Write` to `<worktree>/src/x.ts` (not in `write_paths`) → exit 2.
- `Write` to an absolute path outside the worktree → exit 2.
- Malformed PreToolUse stdin → exit 2 (fail closed).
- Same command with NO policy env → exit 0 (not a fenced context;
  the hook never bricks a worker over our own wiring).

## Acceptance

Pass: every Section 1–2 assertion holds via the real spawn pipeline +
real fence-hook subprocess. Deterministic; no real `claude -p` (only
the subprocess/NDJSON boundary is stubbed).
