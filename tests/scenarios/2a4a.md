# Phase 2A.4a — Production Scenario

Drive the four Maestro-only MCP tools added in 2A.4a (`ask_user`,
`review_diff`, `research_wave`, `global_status`) through an in-process
MCP client against a Symphony orchestrator backed by a real `git`
worktree stack. No `claude -p` subprocess — this scenario is pure
orchestration-layer + real filesystem.

## Given

- A fresh temp git repo with one commit (no `claude` required).
- An orchestrator server created via `InMemoryTransport` with:
  - `initialMode: 'act'`
  - `defaultProjectPath: <temp repo>`
  - a real `WorktreeManager`
  - a **fake** `WorkerLifecycle` that registers `WorkerRecord`s whose
    `worktreePath` points at real on-disk worktrees (so `review_diff`
    can actually read `git diff` output).
  - in-memory `ProjectRegistry`, `QuestionRegistry`, `WaveRegistry`.
- One registered project (`main`).

## When

1. Client calls `ask_user` with a blocking question. Expect `question_id`
   in `structuredContent`; `QuestionStore` now has 1 entry.
2. Client calls `ask_user` again with an advisory question tied to a
   project. Expect two records, one per urgency.
3. Client calls `research_wave` with n=3 and no agenda. The fake
   lifecycle registers 3 worker records pointing at fresh worktrees.
   Expect `wave_id` + 3 `worker_ids` in the response; `WaveStore` has 1
   entry with `size: 3`.
4. Client writes a file into the first research worker's worktree +
   stages it.
5. Client calls `review_diff` against that worker id. Expect the diff
   body to contain the new filename; `files` has one entry with status
   `A`.
6. Client calls `global_status`. Expect totals = `{projects: 1, workers:
   3, active: 3}` and the `main` project bucket at `active: 3`.
7. Client calls `global_status` with `uncommitted: true`. Expect a
   `uncommitted` array entry for the worker that has staged changes
   (`has_changes: true`, `staged: 1`).
8. Client calls `review_diff` with an unknown worker id. Expect
   `isError: true`.
9. Client calls `research_wave` with `n: 1` (under the minimum). Expect
   `isError: true` (schema violation surfaces as an MCP error).

## Then

Observable state:

- `QuestionStore.size()` = 2; blocking + advisory records present.
- `WaveStore.size()` = 1; wave has 3 worker ids that match the
  registered workers.
- `git diff --name-status` run in the worker's worktree shows the new
  file with status `A`, matching `review_diff`'s report.
- `global_status` totals exactly match worker counts; the `uncommitted`
  variant surfaces staged file counts.
- Unknown worker + under-min `n` both return `isError`.

## Cleanup

All worktrees removed via `WorktreeManager.remove` (idempotent on
already-missing dirs). Temp repo removed via `rmSync({ force: true,
recursive: true })`. Orchestrator closed (`server.close()`). No orphan
processes or leftover worktrees.
