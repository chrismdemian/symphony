# Phase 3H.4 — Production scenario

**Goal:** prove the keybind override editor end-to-end through the real
launcher boot — opening the editor from the settings popup, capturing
a new chord, persisting to disk, and resetting back to default.

## Steps

1. `runStart` boots with default config under a temporary
   `SYMPHONY_CONFIG_FILE` sandbox.
2. User opens settings via `/config Enter` (slash-command path —
   matches 3H.1 / 3H.2 / 3H.3 scenarios; Ctrl+, requires kitty CSI-u
   which ink-testing-library can't deliver).
3. User navigates DOWN to the `keybindOverrides` row.
4. Enter pushes the keybind editor popup. The popup lists every
   non-internal registered command (focus cycle, exit, palette, help,
   etc.).
5. User navigates to the `help` row (`app.help` / chord `?`).
6. Enter arms capture mode. Frame switches to the capture view.
7. User presses `F`. `chordFromInput` returns `{kind:'char', char:'F'}`.
   `detectKeybindConflicts` finds no collision (no other command binds
   `F`). `setConfig` writes the override to disk.
8. Disk file now contains
   `keybindOverrides: { "app.help": { kind: "char", char: "F" } }`.
9. Capture popup pops; editor returns to list mode. The `help` row
   shows `F (override)`.
10. User navigates back to the `help` row and presses `r`.
11. `setConfig` strips the entry. Disk file `keybindOverrides` is
    `{}` again.
12. Esc twice — editor closes, settings popup closes, chat placeholder
    returns.

## What this exercises (vs. unit tests)

- The full settings popup → keybindOverrides row → editor push
  → capture push → setConfig → applyPatchToDisk → on-disk JSON path.
- The disk-side `keybindOverrides` write through `applyConfigEdits`'s
  field list (regression for 3H.3 m1 — schema fields must round-trip).
- The reset path: setConfig with a function-patch that removes a key
  from the record.
- The dispatcher receives the override-applied command set (verified
  indirectly via the `(override)` tag rendering).

## Out of scope

- Conflict-error path (covered by KeybindCapturePopup unit tests).
- Modifier-only rejection (covered by chord-from-input unit tests).
- Real worker spawns (mocked through the integration test).
- Cross-process write contention (Phase 5/8 follow-up).
