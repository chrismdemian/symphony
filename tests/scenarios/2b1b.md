# Phase 2B.1b — Worker-metadata persistence (production scenario)

**Goal:** Prove that real `claude -p` worker subprocesses spawned via the
SQLite-backed orchestrator are reflected in SQL, and that a fresh
orchestrator process started against the same DB correctly reconciles
abandoned workers as `crashed`.

This is the authoritative end-to-end test for 2B.1b — distinct from the
integration test which uses fake worker primitives and `InMemoryTransport`.
Here we launch a real `claude -p` subprocess via `WorkerManager.spawn`,
deliberately abandon the orchestrator (without calling its `shutdown`),
open a SECOND orchestrator on the same DB, and verify recovery flips the
worker to `crashed` while preserving its `sessionId`.

## Steps

1. Init a fresh git repo at `<sandbox>/repo` with a single committed
   `README.md`. The default project for the orchestrator points there.
2. Open a `SymphonyDatabase` at `<sandbox>/symphony.db`.
3. Spawn orchestrator A in ACT mode against that DB. Connect a client.
4. Call `spawn_worker` with a trivial prompt. Assert the structured
   response carries an `id` matching `^wk-`.
5. Wait for the live registry's `lastEventAt` to update (or for the
   registry status to flip to `running` via the event tap).
6. Read the SQL `workers` row directly: confirm `status='running'` (or
   `'spawning'`) and `session_id` is non-null.
7. **Abandon orchestrator A** — close the CLIENT only. Do NOT call
   `handle.close()`. Capture the live worker handle so we can clean up
   the orphaned subprocess after the test.
8. Open `SymphonyDatabase` instance #2 on the same file. The first call
   to `startOrchestratorServer` runs `recoverFromStore` automatically.
9. Open orchestrator B against the same DB. The startup hook flips the
   abandoned worker's row from `running` → `crashed`, stamps
   `completed_at`, preserves `session_id`.
10. Call `list_workers` via orchestrator B. The crashed worker appears
    with `status='crashed'` and `sessionId` matching what was persisted
    pre-abandonment.
11. Call `global_status` with `uncommitted: false` — verify the per-project
    bucket counts the crashed worker in `failed` and `total: 1`.
12. Cleanup: `record.worker.kill('SIGKILL')` on the orphan, await exit,
    close orchestrator A (idempotent), close orchestrator B, close DB,
    rm sandbox.

## Observable-only assertions

- After `spawn_worker`: SQL row exists with status in {`spawning`, `running`}.
- After abandon → recover: SQL row status is `crashed`, `session_id` preserved.
- `list_workers` returned by orchestrator B shows the crashed worker
  with both `id` and `sessionId` (so a future `resume_worker` call can
  pick up the session).
- `global_status` per-project totals reflect the persisted-only crashed
  worker (live registry on B is empty until the user resumes).

## What this scenario proves

- `WorkerRegistry.register` writes through to `SqliteWorkerStore` via
  the live `claude -p` spawn path (Phase 1B + 1C + 2A wiring intact).
- The schema-contract validator accepts the workers table at startup.
- `createWorkerLifecycle.recoverFromStore()` correctly reconciles
  abandoned workers when called by `startOrchestratorServer` at boot.
- `mergeLiveAndPersisted` surfaces persisted-only crashed workers
  via `list_workers` and `global_status` for the "where was I?" UX.
- `sessionId` survives the orchestrator process boundary, enabling
  user-driven `resume_worker` post-recovery.
