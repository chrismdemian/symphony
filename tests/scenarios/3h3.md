# Phase 3H.3 ‚Äî Production scenario

**Goal:** prove the settings popup's notifications + awayMode toggles
end-to-end through the real launcher boot, and that the awayMode
true‚Üífalse transition triggers the RPC `flushAwayDigest` call.

## Steps

1. `runStart` boots with default config (notifications disabled,
   awayMode false) under a temporary `SYMPHONY_CONFIG_FILE` sandbox.
2. User opens settings via `/config Enter` (slash-command path ‚Äî
   matches 3H.1 / 3H.2 scenarios; Ctrl+, requires kitty CSI-u which
   ink-testing-library can't deliver).
3. User toggles `notifications.enabled` on (Space). Disk reflects.
4. User toggles `awayMode` on (Space). Disk reflects. The TUI's edge
   detector does NOT fire (`prev=false ‚Üí cur=true` is not the trigger).
5. User toggles `awayMode` back off (Space). The TUI's
   `useEffect(awayMode)` detects the `true ‚Üí false` edge and invokes
   `rpc.call.notifications.flushAwayDigest()`. Asserted via spy.
6. `notifications.enabled = false` again (Space toggle) ‚Äî disk
   persists. No further RPC calls.
7. Esc closes the popup, chat placeholder returns.

## What this exercises (vs. unit tests)

- The full settings popup ‚Üí ConfigProvider.setConfig ‚Üí applyPatchToDisk
  serialization queue.
- The `App.tsx` awayMode `useEffect` initial-mount no-fire guard +
  edge detection.
- The `notifications.flushAwayDigest` RPC procedure as it ships
  through the typed router.
- The disk persistence + Zod salvage of the new top-level `awayMode`
  field.

## Out of scope

- Real `spawnToast` (mocked dispatcher). The cross-platform shim is
  unit-tested separately.
- Real worker exits (mocked through the integration test).
