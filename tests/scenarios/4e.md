# Phase 4E scenario — worker structured completion: display + open_questions routing

## Goal

End-to-end verification that a completed worker's structured completion
report flows through the real consumer paths: the report (with a
`display` spec and `open_questions`) reaches the worker buffer via the
real stream-parser logic and the real lifecycle event tap; `open_questions`
are routed into the real `QuestionRegistry` as advisory, auto-acknowledged
entries (rule #7); and the real `get_worker_output` MCP tool renders the
full report — including `open_questions` content — so Maestro can read
it without resuming the worker.

Ground Truth Is Observable: the report event is constructed by the REAL
`scanForCompletionReport` (the exact parser logic the stream-parser
uses), flowed through the REAL `createWorkerLifecycle` tap + exit hook
into the REAL `WorkerRegistry` buffer and REAL `QuestionRegistry`. We
assert on emitted/stored bytes and records — never on the semantic
trustworthiness of the worker's self-reported `audit` / `tests_run`.

## Given

- A real `WorkerRegistry`, real `QuestionRegistry`, real
  `createWorkerLifecycle` wired with the SAME `onWorkerStatusChange`
  closure shape `server.ts` uses (it calls `routeWorkerOpenQuestions`).
- The worker spawned through the REAL `makeSpawnWorkerTool` handler
  (the MCP dispatch entrypoint Maestro uses).
- The OS boundary (the `claude` subprocess + NDJSON transport) is the
  only stub: an `EmittingStubWorker` whose `events` yield the
  `structured_completion` event the real parser would emit
  (`scanForCompletionReport` on a real ` ```json ` completion fence),
  then it exits `completed` after the buffer has the event.

## When / Then

### Section 1 — completion report reaches the buffer (parser → tap → buffer)

Spawn the worker; it emits a final message containing the verbatim
worker-common-suffix JSON contract with `display` (a Card spec) and two
`open_questions`, then exits.

- `get_worker_output` (real tool) text contains the enriched completion
  line `[completion] audit=PASS did=… open_questions=2 …`.
- The text contains the `open_questions:` detail line with BOTH question
  strings verbatim (rule #7: Maestro can read them).
- The buffered `structured_completion` event's `report.display` deep-
  equals the input Card spec (advisory field carried end-to-end).

### Section 2 — open_questions routed to the 3E subsystem (rule #7)

- The real `QuestionRegistry` contains exactly the two open_questions,
  each `urgency: 'advisory'`, `workerId` = the spawned worker id,
  `answered: true`, `answer` = the `OPEN_QUESTION_ACK` sentinel,
  `context` mentioning the worker's feature intent + "not acted on".
- `questionStore.list({ answered: false })` is empty — the advisory
  entries never block the 3E popup nor inflate the StatusBar `Q:`
  count (which both poll `answered:false`).

### Section 3 — advisory degradation is non-fatal

A second worker emits a report whose `display` is a bare string (not a
spec) plus one open_question.

- `get_worker_output` still renders the full textual completion line
  (`audit=…`), and the open_question is still routed advisory +
  auto-acknowledged. A malformed `display` never blocks parsing,
  routing, or the audit fields (advisory contract).

## Acceptance

Pass condition: all Section 1–3 assertions hold via real lifecycle +
real registries. Deterministic; no real `claude -p` required (the
report event is produced by the real `scanForCompletionReport`, the
exact logic the stream-parser uses — only the subprocess/NDJSON
transport is stubbed, the standard Track-1 boundary).
