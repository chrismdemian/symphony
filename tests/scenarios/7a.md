# Phase 7A scenario — plugin tool callable end-to-end with non-defeatable audit

## Goal

End-to-end verification that an enabled plugin's tool, surfaced as a namespaced
proxy in Symphony's MCP server, is callable by a Maestro-equivalent MCP client
AND that every call lands in the audit log BEFORE dispatch — the security
mandate from the PLAN's plugin envelope.

## Tested paths

1. **Two-gate activation.** Orchestrator boots with `options.plugins.enabled`
   (the `--plugins` flag Maestro's MCP child carries) AND the `pluginsEnabled`
   config master switch ON. The real `echo` plugin subprocess
   (`tests/fixtures/plugins/echo/server.mjs`) is spawned via the production
   StdioClientTransport.

2. **Tool discovery → namespaced registration → MCP listTools.** A real MCP
   `Client` connected to the orchestrator sees `echo__ping`, `echo__echo`, etc.
   in `listTools()` (double-underscore namespace — `:` is not an Anthropic
   tool-name character).

3. **Maestro-equivalent call → proxy → real subprocess → result.** The client
   calls `echo__ping` and `echo__echo`; the call flows through `wrapToolHandler`
   (scope/capability/safety/audit) and the proxy forwards to the plugin
   subprocess, returning `pong` / `echo: <text>`.

4. **Non-defeatable audit BEFORE dispatch.** After the calls, the audit store
   contains `tool_called` rows for `echo__ping` and `echo__echo` — the same
   chokepoint every built-in tool flows through, now covering plugin tools.

5. **Clean teardown.** `handle.close()` shuts the plugin subprocess down with no
   leaked process.
