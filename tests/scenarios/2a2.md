# Phase 2A.2 — Production Scenario

Drive the full worker-lifecycle tool chain against a real `claude -p`
subprocess. An in-process MCP client (`InMemoryTransport`) connects to a
Symphony orchestrator server. The client calls `spawn_worker` in ACT mode
with a trivial prompt; Symphony composes `WorktreeManager.create` + a real
`WorkerManager.spawn(claude -p)` in the spawned worktree. The lifecycle
event tap feeds the worker's stream into the registry's circular buffer,
and the client then drives `list_workers`, `get_worker_output`, and
`kill_worker` through the SDK to observe the full shape Maestro will see.

## Given

- A `claude` binary resolvable via PATH (`claude --version` exits 0). If
  missing, this scenario **skips with a warning** so CI without claude still
  passes.
- A fresh temp git repo with one commit. The orchestrator's
  `defaultProjectPath` points at this repo. `WorktreeManager` uses it
  unpooled.
- Orchestrator started in ACT mode via `InMemoryTransport` so the test is
  a single-process harness (no `symphony mcp-server` subprocess).

## When

1. Build `dist/index.js` (produced by `pnpm build` at CI bootstrap).
2. Initialize a temp repo with one commit.
3. Start the orchestrator server in ACT mode against the repo.
4. Client calls `tools/call spawn_worker` with a minimal prompt that asks
   the worker to print one line of text and stop. The worker should
   produce a `result` event and exit cleanly.
5. While the worker is alive, the client calls `list_workers` and expects
   the spawned id to appear with `status='running'` or `status='spawning'`.
6. After waiting briefly, the client calls `get_worker_output` for the
   worker id; the returned event buffer contains at least one
   `system_init` entry.
7. The client waits for the worker to complete naturally (its prompt told
   it to exit) using the orchestrator's `workerRegistry.get(id)!.worker.waitForExit()`.
8. The client calls `list_workers` again and expects the worker's status
   to be `'completed'`.

## Then

Observable state:

- `spawn_worker` returns `{ id, worktreePath, role, featureIntent, … }`
  with `id` matching `/^wk-/` and a worktree path under
  `<repo>/.symphony/worktrees/<id>`.
- At least one event is captured in the worker's circular buffer.
- `list_workers` after exit shows `status='completed'`.
- No `parse_error` appears in the captured stream.

## Cleanup

Temp repo removed. Orchestrator closed (shutdown drains the worker
lifecycle). No orphan claude processes.
