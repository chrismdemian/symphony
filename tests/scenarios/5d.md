# Phase 5D — Production scenario: `set_active_project` routes omitted-`project:` calls correctly

## Given

- Two real git project directories at `<sandbox>/projA/` and `<sandbox>/projB/`,
  each with `git init` + initial commit.
- A real on-disk SQLite DB.
- Both projects registered via `SqliteProjectStore.register` (same call path
  `startOrchestratorServer` uses at boot).
- A real `~/.symphony/config.json` (sandboxed via `SYMPHONY_CONFIG_FILE` env)
  with no `activeProject` field — the cursor starts null, the resolver falls
  back to `defaultProjectPath` (= `projA`'s path).
- A live `startOrchestratorServer` with both projects registered and a
  caller-supplied `completionsBroker` so the scenario can verify chat-row
  signaling end-to-end.

## When

A user-style scenario unfolds against the live wiring:

1. The user (via Maestro) calls `set_active_project(project_name: "projb")`.
2. A subsequent `create_task(project: "projb", description: "...")` lands a
   task on projB. The task store records it under `projB`'s `id`.
3. A subsequent `create_task` with NO `project:` arg lands on the active
   project — `projB`. The task store records it under `projB`'s `id`.
4. The user calls `set_active_project(project_name: "(none)")` to clear.
5. A subsequent `create_task` (no `project:` arg) falls back to the
   `defaultProjectPath` = `projA` and lands on projA's `id`.
6. `~/.symphony/config.json` reflects `activeProject: undefined` after the
   clear (the clear is persisted).

## Then

1. **After step 1** — `set_active_project` returns `isError: false`, the
   structured response carries `active.name === "projb"` and
   `active.path === <projB resolved path>`.
2. **After step 1** — the completionsBroker received exactly one summary
   with `headline === "Active project → projb"` and `statusKind: "completed"`.
3. **After step 2** — the new task's `projectId` is `projB`'s store id.
4. **After step 3** — even with `project:` omitted, the task lands on
   projB (the active cursor wins over `defaultProjectPath`).
5. **After step 4** — the broker received a second summary with
   `headline === "Active project cleared (was projb)"`.
6. **After step 5** — the omitted-project task lands on projA's id (boot
   default takes over once the cursor is null).
7. **After step 6** — `loadConfig()` from a fresh read returns
   `activeProject === undefined`.

## Notes

- Tests `create_task` rather than `spawn_worker` because spawn would
  require booting a real `claude` subprocess. The same resolver path
  (`resolveProjectPath` in `server.ts`) feeds both, so create_task is
  a sufficient observable.
- The integration test (`5d-active-project.integration.test.ts`)
  covers more edge cases (rejection of unknown projects, idempotency,
  boot-time restoration) at the API level; this scenario verifies the
  end-to-end shape with disk + SQLite + a real registered server.
