# Phase 3C scenario — Worker panel through the launcher

## What this exercises

A full launcher boot (`runStart` with fakes for Maestro, hook server,
RPC, worker manager) where the fake RPC exposes a mutable workers
list. The scenario asserts:

1. Initial render shows the empty-state hint when the list is empty.
2. After the RPC poll picks up two workers, both rows appear with
   their instrument names.
3. After one worker flips to `completed`, the rendered frame includes
   the `✓` glyph (visual evidence of the status flip).
4. The workers panel content sits inside the bordered "Workers"
   panel.

This proves the data-hook → registry → panel chain works end-to-end
through the same code path the production launcher uses, not just
isolated unit fakes.

## Coverage limits

- The 1-second `useWorkers` poll is real wall-clock here. The
  scenario inserts `setTimeout` waits ≥1.1s so the next tick fires.
  Faking timers across the launcher bootstrap deadlocks (see 3B.3
  scenario rationale).
- `workers.kill` is exercised in `tests/ui/panels/workers/WorkerPanel.test.tsx`
  with stdin keystroke injection. The launcher scenario doesn't
  re-test it — `useInput` keystroke routing is identical to 3B.3's
  status-line scenario.

## Why fileParallelism: false

`vitest.scenarios.config.ts` already enforces this for the whole
suite. 3C doesn't spawn `claude` but inherits the global setting
because real-claude scenarios race on `~/.claude.json`.
