# Phase 3K — Worker Completion Summaries production scenario

## What this scenario verifies

End-to-end through `runStart`:

1. The TUI subscribes to the orchestrator's `completions.events` topic
   on mount.
2. When the orchestrator publishes a `CompletionSummary` (simulated
   here by the fake RPC's subscribe handler firing the listener), the
   chat panel renders a new system row.
3. The system row carries the status icon (`✓`), the resolved worker
   name, the project name in parens, the formatted duration, and the
   summary headline body line.

## Inputs

- Fake Maestro (no real Claude subprocess).
- Fake RPC client whose `workers.list` returns one worker (so the TUI's
  instrument allocator assigns it `Violin`) and whose `subscribe` for
  `completions.events` immediately invokes the listener once with a
  canned `CompletionSummary` payload.
- Real keybind dispatcher and TUI tree under `runStart`.

## Acceptance

- After mount + first poll cycle settles, stdout contains:
  - `✓` glyph
  - `Violin` (TUI-resolved name from instrument allocator, NOT the
    server-supplied slug `worker-wk-1`)
  - `(demo)` project label
  - The summary headline `wired the friend system endpoints`
- The fake RPC `subscribe` was called with topic `'completions.events'`.
