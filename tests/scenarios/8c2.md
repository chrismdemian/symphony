# Scenario 8C.2 — GitHub Issues sync (connector + comment/close writeback)

**Goal:** prove the GitHub connector end-to-end against a real orchestrator
server, a real `GitHubConnector` (over the injected `GitHubClientLike` seam, no
live API), and temp-file SQLite — plus the persistence-across-a-fresh-store leg
and the on-the-wire comment + close writeback.

**Setup**
- Temp-file SQLite db.
- A fake `GitHubClientLike` returning a mix of open + closed issues across two
  repos, plus PRs (issues carrying a `pull_request` field) — though the client
  fake returns already-filtered nodes; the closed/terminal skip is exercised.
- A real `GitHubConnector` built on that client, injected via `githubConnector`.
- Two registered projects (`acme/app`, `acme/api`) so routing-by-repo resolves.
- Server booted at autonomy tier 2 (`sync_github` is secrets/network/external-visible).

**Steps & expectations**
1. `sync_github` creates one pending task per open issue, routes by `owner/repo`,
   and **skips** issues already closed (`skippedDone`).
2. A second `sync_github` creates nothing (idempotent — `skippedExisting`).
3. External links persist: a fresh `SqliteExternalLinkStore` over the same db
   file lists every imported `owner/repo#number` under `source='github'`.
4. Completing a task fires the writeback hook → the real connector posts a
   comment then closes the issue (the client records both calls).
