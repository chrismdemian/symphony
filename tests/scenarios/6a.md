# Phase 6A — Production scenario: `symphony voice diagnose` drives the real bridge end-to-end

## Given

- `~/.symphony/voice-env/` exists with `silero-vad`, `sounddevice`, `numpy` installed
  (created by `symphony voice install`).
- `tests/fixtures/voice/diagnose-3s.pcm` exists — a deterministic
  3.3 s int16 mono 16 kHz PCM fixture containing two speech-shaped
  bursts separated by silence (see `tests/fixtures/voice/generate.mjs`
  for the recipe; checksums committed at `CHECKSUMS.txt`).

## When

Invoke `runVoiceDiagnose({ format: 'json' })` from a test harness — the same
function the `symphony voice diagnose --json` CLI command calls. The
runner:

1. Resolves the voice venv via `resolveVoiceEnv()`.
2. Spawns the Python bridge via `VoiceBridge.start({ inputMode: 'stdin-pcm', ... })`
   with the real venv's python interpreter.
3. Awaits the `ready` event (Silero model load completes).
4. Pipes `diagnose-3s.pcm` chunk-by-chunk through the bridge's stdin.
5. Closes stdin → bridge drains, emits `shutdown_ack`, exits 0.
6. Tallies `speech_start` / `speech_end` events.

## Then

Observable assertions (no worker self-reports — every assertion is on
process output or the structured result):

1. `result.ok === true`.
2. `result.exitCode === 0`.
3. `result.speechSegments >= 1` (Silero output is sensitive to version
   so we don't pin exactly 2 — but the fixture contains two distinct
   bursts and any working pipeline produces ≥ 1 segment).
4. `result.events` contains at least one `{ type: 'ready', backend: 'stdin-pcm', sampleRate: 16000 }`.
5. `result.events` contains at least one `{ type: 'speech_start' }` and
   at least one `{ type: 'speech_end' }`.
6. The bridge subprocess exited cleanly (no SIGTERM / SIGKILL escalation).
7. Total wall-clock duration < 60 s (Silero model load + 3.3 s of audio
   chunked at ~25 ms intervals; budget headroom for cold-start variance).

## Notes

- Skip-gracefully when `~/.symphony/voice-env/` is missing or
  silero-vad isn't installed. The skip is loud (vitest test marked
  `skipped`, `console.warn` line emitted) so CI failures are visible.
- The fixture's two-burst structure is deliberate: Phase 6B's STT
  integration will assert that each burst transcribes to non-empty
  text. Don't reshape this fixture without updating 6B's scenario.
- The diagnose CLI command is the scenario gate. Production users
  invoke `symphony voice diagnose` after installing voice — this same
  flow is what the gate exercises.
