# Phase 8B scenario — Obsidian sync end-to-end (real DB, real server, real vault)

## Goal

End-to-end verification of the in-tree Obsidian connector against a real
orchestrator boot backed by a temp-FILE SQLite database (not `:memory:`), so
external-link persistence is proven across a fresh store instance — the
restart-survives-dedup guarantee that makes re-syncing safe.

Unlike 8A (where a fake Notion connector stands in for the live API), Obsidian's
"API" IS the filesystem, so this scenario uses a REAL `ObsidianConnector` over a
REAL temp vault: real `startOrchestratorServer`, real `SqliteTaskStore` /
`SqliteExternalLinkStore`, real `gray-matter` frontmatter parsing, real markdown
files on disk, real MCP `Client` dispatch through the capability shim (tier 2,
since `sync_obsidian` declares `external-visible`).

## Tested paths

1. **Tool surface + capability gating.** `sync_obsidian` is registered only
   because a connector is injected, and a tier-2 MCP `Client` can call it
   (`external-visible` requires tier >= 2).

2. **Create + route + link.** A vault with two notes (frontmatter `project:`
   routing two open tasks to two different projects, plus a `[x]` done line) →
   two Symphony tasks created in SQLite, two external links written, the done
   line skipped (`skippedDone`).

3. **Idempotent re-sync.** A second `sync_obsidian` over the same vault creates
   zero tasks (`skippedExisting`) — dedup by the stable file#locator external id.

4. **Persistence across a fresh store.** After the sync, a brand-new
   `SqliteExternalLinkStore` opened over the SAME db file still resolves the
   links — proving re-sync after a Symphony restart stays idempotent.

5. **Bidirectional checkbox writeback.** Driving an imported task to `completed`
   fires the `onTaskStatusChange` fan-out → the Obsidian writeback hook → the
   connector flips `- [ ]` → `- [x] ✅ <date>` in the SOURCE markdown file. The
   sibling task line in the same file is untouched.

## Coverage notes

- The parser's grammar (status chars, emoji + Dataview metadata, locator
  precedence, checkbox rewrite, fenced-code skipping) is covered by
  `tests/integrations/obsidian-parser.unit.test.ts`.
- The connector's read/writeback over a fake vault is covered by
  `tests/integrations/obsidian-connector.unit.test.ts`.
- The live chokidar watcher (debounce, loop-safety, ignore filters) is covered
  by `tests/integrations/obsidian-watcher.unit.test.ts` (deterministic, injected
  factory) and `tests/integration/8b-obsidian-sync.integration.test.ts` (real
  chokidar against a temp vault).
- The `:memory:` variant of paths 1–3 + 5 lives in
  `tests/integration/8b-obsidian-sync.integration.test.ts`; this scenario adds
  the temp-FILE persistence leg (path 4).
